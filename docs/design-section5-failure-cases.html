2. 演示脚本 1：成功案例

✅ 成功案例

场景：验证 ECS 迂曲度公式 λ = √(D/D*)

1
论文原文（第 3 页，Volume Fraction 章节）：
"In the cortex, the tortuosity λ was found to be approximately 1.6, with D = 7.6×10⁻⁶ cm²/s in free medium and D* ≈ 2.9×10⁻⁶ cm²/s in the extracellular space."
2
Agent Extract 输出：
claim C-001: "皮层中 λ ≈ 1.6 (D=7.6e-6, D*=2.9e-6 cm²/s)"
extraction_method: exact_quote
source: page=3, section="Volume fraction measurements"
numerical_statement N-001: formula_nl="λ = √(D/D*)", params={D:7.6e-6, D_star:2.9e-6}, reported=1.6
3
Agent Compute 输出：
公式转译: "λ = √(D/D*)" → python_expr = "sqrt(D / D_star)"
安全执行: sqrt(7.6e-6 / 2.9e-6) = 1.618...
结果对比: |1.618 - 1.6| / 1.6 = 1.1% < 5% → passed ✓
4
ARM 最终记录：
claim C-001 status: supported, dry_run: passed
"Agent 不仅提取了结论，还用代码验证了数据一致性——误差仅 1.1%。"

3. 演示脚本 2：失败案例 — insufficient_data → review_required

❌ 失败案例（推荐 PPT 必演）

场景：论文提出定量结论但缺少验证所需参数

1
论文原文（第 8 页，Pathological Changes 章节）：
"Following the induction of ischemia, the extracellular volume fraction decreased significantly, reaching approximately 30% of its baseline value within 5 minutes."
2
Agent Extract 输出：
claim C-004: "缺血后 ECS 体积分数 α 在 5 分钟内降至基线的约 30%"
extraction_method: exact_quote
numerical_statement N-004: formula_nl="α_ischemia = α_baseline × 0.30", params={}, reported=null
3
Agent Compute 尝试 dry-run：
需要参数: baseline_α, ischemia_α, time_course_data
可获取: 仅 relative_change = 30%，无绝对值
→ insufficient_data
4
Agent 行为（关键演示点）：
标记 review_required
原因: "论文仅报告了 α 的相对变化（30%），未提供基线 α 绝对值、完整时间序列数据或统计检验结果，无法进行数值验证。"

Agent 没有：
✗ 编造一个 baseline_α 值来强行计算
✗ 用其他论文的数据填补缺口
✗ 假装验证通过
5
ARM 最终记录：
claim C-004 status: partial, dry_run: insufficient_data
"Agent 正确识别了数据不足的情况，标记 review_required 而非编造数据。
这正是 ARM 的核心价值——区分'论文说了什么'和'我们能验证什么'。"

4. 演示脚本 3：失败案例 — Hallucination 检测

❌ 失败案例（推荐 PPT 必演）

场景：LLM 生成的 quoted_text 在原文中无法匹配

1
Extract 阶段 LLM 返回：
claim C-005: "ECS 体积分数在睡眠期间增加约 60%"
extraction_method: exact_quote
quoted_text: "The ECS volume fraction increases by approximately 60% during sleep compared to the awake state."
2
Validate 阶段 — Hallucination 检测：
精确匹配: "...increases by approximately 60%..." 在原文第 5 页中 未找到
Fuzzy 匹配: token overlap = 0.45 < threshold 0.60 → 未通过
→ hallucination_flag
3
Agent 行为：
标记 review_required
将 extraction_method 从 exact_quote 降级为 review_required
原因: "quoted_text 在原文指定页面中无法匹配。原文实际表述为 '...a 40-60% increase in the extracellular space was observed...'——LLM 对数值进行了简化并改写了措辞。"
4
ARM 最终记录：
claim C-005 extraction_method: review_required
hallucination_flag: true
"防幻觉机制在工作：quoted_text 无法在原文中定位时，自动触发 review_required。
这让 ARM 的 provenance 是可以真正信赖的。"

6. 安全规则清单

#规则类型实施位置违规时行为
S1 不编造数据：当参数缺失时，不生成虚构数值来填充 输入/输出 Extract prompt + Compute 阶段 标记 insufficient_data + review_required
S2 模型推断不冒充原文：所有 LLM 推断必须标记 llm_inferred 输出 Extract prompt + Validate R6 Validate 阶段检测到未标记推断 → review_required
S3 不伪造引用：quoted_text 必须能在原文中定位 输出 Validate R4 Hallucination 检测失败 → review_required
S4 不提供医学诊断：ARM 中的疾病关联结论仅限于科研证据整理 输出 Extract prompt Prompt 中明确约束"不提供自主诊断或处方"
S5 Dry-run 安全隔离：计算代码不能访问文件/网络/系统 工具 dry_run_calc 沙箱 沙箱白名单拦截 + calculation_error
S6 拒绝真实患者信息：输入检查，拒绝含 PHI 的内容 输入 Orchestrator 入口 终止运行，提示"检测到可能的患者信息"

7. PPT 演示策略

演示环节案例时长展示内容
端到端成功 成功案例 ~3 分钟 上传论文 → Streamlit 展示 ARM → 高亮 claims + evidence 关联 → 展示 dry-run passed → 下载 YAML
失败路径 1 失败案例 2 ~2 分钟 展示论文中"α 降至 30%"的结论 → Agent 尝试验证 → insufficient_data → review_required 标记 → ARM 中明确记录
失败路径 2 失败案例 4 ~2 分钟 展示 LLM 生成的 quoted_text → Hallucination 检测未通过 → extraction_method 降级 → 强调 provenance 防伪机制
🎯 演示台词脚本建议：

成功："Agent 不仅读懂了论文，还自动发现了结论中隐含的计算关系——迂曲度 λ = √(D/D*)，并用代码验证了数据一致性。这就是 ARM 的'可执行性'。"

失败："当论文提出了一个定量结论但缺少验证所需的具体数值时，Agent 正确地标记了 review_required，而不是编造数据来通过验证。这体现了 ARM 的'可追溯性'和安全性——区分已知和未知。"
